## 1. 产品宗旨

### 1.1 问题陈述

人们想要养成好习惯，却很难坚持下去——这不是因为他们缺乏意愿，而是因为他们缺乏**系统**。

具体而言，我们的潜在用户正面临三类核心困境：

1. **启动容易，坚持困难**——用户每次下定决心时都充满热情，制定完美计划，但执行中一旦某天"断链"，就会产生"已经失败了"的心理暗示，进而全盘放弃。他们缺少的不是计划能力，而是**容错与韧性机制**。

2. **时间碎片化，习惯无处安放**——在工作、学业和生活的挤压下，用户很难找到固定的习惯执行窗口。当计划要求"每周一三五健身"而周三加班到深夜时，刚性计划直接失效，缺乏弹性调整的空间，用户只能选择放弃。

3. **缺乏反馈，动力枯竭**——习惯养成的回报是延迟的：坚持早睡三天，身体几乎感觉不到变化。没有即时、可见的进步信号，用户的动力在最初的热情消退后迅速枯竭。

**一句话概括**：用户的核心问题不是"不知道该做什么"，而是"做不到持续做"——他们在从意愿到行动、从行动到坚持之间，缺少一座可依赖的桥梁。

### 1.2 目标用户

MVP版本主要服务于以下两类核心用户：

#### 主要用户：生活节奏不稳定、渴望重建秩序的年轻人

以**林小禾**（21岁大学生）和**赵一鸣**（26岁职场人）为代表——他们共同的特征是：

- 有明确的改变意愿（考研、健康预警等强驱动力）
- 生活节奏不稳定，习惯执行的时间窗口经常被打乱
- 过去多次尝试养成习惯但反复失败，对自身"意志力"产生怀疑
- 需要简单、低门槛的工具——太复杂的设置本身就是放弃的理由

> **MVP不优先服务**像苏铭远那样对数据有深度量化需求的用户。这类用户需要的数据导出、关联分析、自定义指标等功能，将作为高级功能在后续版本中迭代，而非MVP阶段的核心关注点。

### 1.3 核心价值

我们的App通过三个核心机制，帮助用户跨越"从意愿到坚持"的鸿沟：

| 核心价值 | 解决的痛点 | 实现方式 |
|----------|-----------|----------|
| **让坚持被看见** | 缺乏反馈，动力枯竭 | Streak连续打卡 + 日历视图，将抽象的"坚持"转化为可见的数字和图案 |
| **让计划有弹性** | 计划崩盘后放弃；时间碎片化与缺乏弹性 | 灵活频率设置（如"每周3次"）+ 断链保护，保持习惯连续性不被一次意外中断 |
| **让起步更简单** | 启动容易坚持困难；同时想改太多事 | 一键打卡极简交互 + 习惯分类分组，降低执行摩擦力和认知负担 |

**一句话价值主张**：

> 习惯追踪器帮助生活节奏不稳定的年轻人，用弹性计划、可视化反馈和极简操作，把"想坚持"变成"能坚持"。

## 2. 用户画像

### 2.1 画像一：压力很大的在校大学生

**姓名与照片描述**：林小禾——一位戴着黑框眼镜、扎着低马尾的女生，照片中她坐在大学图书馆靠窗的位置，桌上摊着课本和一杯已经凉掉的咖啡，表情略显疲惫但眼神认真。

**个人简介与背景故事**：林小禾是某985高校大三心理学专业的学生。大一、大二时她成绩优异，但进入大三后面临考研准备、专业课加重和学生会工作三重压力，生活逐渐失控——经常熬夜到凌晨2点，早上翘掉8点的课，靠外卖度日，运动几乎为零。她曾多次尝试"重新做人"：买过手账、下载过番茄钟App、甚至制定了精确到半小时的日程表，但每次都坚持不到一周就回到原样。她开始怀疑自己是不是"意志力天生就比别人弱"。

| 项目 | 详情 |
|------|------|
| 年龄 | 21岁 |
| 职业 | 大三在读学生 |
| 地点 | 武汉 |
| 生活状态 | 独居校内单人间，社交圈以同学为主 |

**目标**：
1. 重建稳定的作息规律——每天11:30前入睡、7:00起床，不再靠熬夜"还债"。
2. 每天保持至少1小时的高效学习时段——不被手机和社交媒体打断。

**痛点与挫败**：
- 制定了完美的计划，但执行第一天就因为"突发作业"或"同学叫去吃饭"而全盘崩溃，之后干脆放弃。
- 缺乏即时反馈——坚持了三天早睡，但感觉"没什么变化"，动力迅速消退。
- 习惯之间没有关联，总是想同时改变5件事，结果一件都没坚持。

**动机**：
- 看到室友每天6:30起床去跑步，内心既羡慕又不甘。
- 考研是她的核心目标，她深知以现在的状态"绝对考不上"，这种焦虑是最强大的驱动力。
- 如果能看到自己"连续坚持了7天"的记录，她会不想让这个数字归零。

**一句引言**：*"我不是不想改变，我是每次刚开始就全盘崩掉，然后觉得自己没救了。"*

### 2.2 画像二：工作繁忙的年轻职场人

**姓名与照片描述**：赵一鸣——一位穿着简洁衬衫的男生，照片中他站在地铁车厢里，单手抓着扶手，另一只手拇指快速滑动手机屏幕处理工作消息，眼下有淡淡的青黑。

**个人简介与背景故事**：赵一鸣在一家互联网公司做产品经理，入职两年，正处于职业上升期。他的日常是这样的：早上9点到公司，晚上9点后才能离开，周末还经常被拉进项目群。工作填满了他几乎所有清醒的时间。他曾是个热爱生活的人——大学时坚持健身、周末去徒步、还自学过吉他。但工作后，这些习惯一个个"自然死亡"。他不是没试过恢复：办了健身卡（去了3次）、买了吉他（在角落积灰）、下载了读书App（开了会员但一本没读完）。每次失败后他都用"太忙了"安慰自己，但内心越来越觉得生活在被工作吞噬。

| 项目 | 详情 |
|------|------|
| 年龄 | 26岁 |
| 职业 | 互联网产品经理 |
| 地点 | 深圳 |
| 生活状态 | 独居，合租两居室，单身 |

**目标**：
1. 每周至少运动3次——不求回到大学时的身材，但至少别再"发福"。
2. 每天留出30分钟完全属于自己的时间——用来阅读或学吉他，而不是刷短视频。

**痛点与挫败**：
- 时间极度碎片化，很难找到完整的"习惯时间窗口"——好不容易挤出30分钟，一个工作电话就打断了。
- 缺乏弹性——健身计划写的是"每周一三五"，但周三加班到10点就彻底泡汤，没有"补卡"机制让他觉得计划已经失败。
- 一个人坚持太难——身边同事的生活方式都差不多（加班+外卖+躺平），没有同辈压力反而有"合谋放弃"的氛围。

**动机**：
- 体检报告上的脂肪肝预警让他第一次真正感到恐慌。
- 偶尔翻到大学时的照片，那个精神饱满的自己让他既怀念又不甘。
- 如果有一个社群或伙伴和他一起坚持，他的执行力会大幅提升。

**一句引言**：*"我知道该运动、该读书，但每天加完班回到家，我只想瘫着，什么都不想动。"*

### 2.3 画像三：对数据有要求的健身爱好者

**姓名与照片描述**：苏铭远——一位身材精干的男生，照片中他刚从健身房出来，穿着速干T恤，手臂肌肉线条分明，左手握着摇摇杯，右手在手机上查看训练记录，表情专注而满足。

**个人简介与背景故事**：苏铭远是一名软件工程师，也是一名有3年经验的健身爱好者。他追踪每一次训练的组数、重量、休息时间，用MyFitnessPal记录每一餐的宏量营养素，用Apple Watch监测心率和睡眠。他的生活高度结构化：周一推日、周三拉日、周五腿日，饮食按周期调整碳水量。但他有一个困扰：健身之外的生活习惯（睡眠、冥想、饮水、补剂服用）散落在不同的App和备忘录里，没有一个统一的视图。他想要一个能像追踪健身数据一样精确追踪所有习惯的工具——要能导出数据、能看趋势图、能发现"睡眠不足时深蹲表现下降"这种关联。

| 项目 | 详情 |
|------|------|
| 年龄 | 29岁 |
| 职业 | 软件工程师 |
| 地点 | 杭州 |
| 生活状态 | 与女友同居，生活规律且自律 |

**目标**：
1. 将健身之外的习惯（睡眠、饮水、冥想、补剂）纳入统一的数据追踪体系。
2. 发现习惯之间的关联规律——比如"连续3天睡眠不足7小时时，饮水量和冥想完成率是否下降"。

**痛点与挫败**：
- 现有的习惯追踪App太"轻"——只有打卡✅/❌，没有量化数据，无法做深度分析。
- 数据孤岛——健身数据在Strong，饮食在MyFitnessPal，睡眠在Apple Health，习惯在备忘录，无法交叉分析。
- 缺乏导出和自定义报表能力——他想生成月度复盘报告，但现有App只提供固定的简单图表。

**动机**：
- 他是"数据即力量"的信徒——任何能被量化的东西，就能被优化。
- 在健身社群中分享自己的数据和复盘文章，获得的认可和关注是他持续追踪的社交驱动力。
- 对"系统化"本身有近乎偏执的热爱——把混乱变成有序让他感到愉悦。

**一句引言**：*"如果一件事值得做，它就值得被量化。光打个勾不够，我要看到数据说话。"*

### 2.4 三个画像的核心差异总结

| 维度 | 林小禾（大学生） | 赵一鸣（职场人） | 苏铭远（健身爱好者） |
|------|------------------|------------------|----------------------|
| 核心需求 | 从0到1建立习惯 | 在有限时间中维持习惯 | 精确量化与深度分析习惯 |
| 最大障碍 | 计划崩盘后放弃 | 时间碎片化与缺乏弹性 | 数据分散无法关联分析 |
| 关键功能偏好 | Streak连续打卡、习惯堆叠 | 弹性频率、社交监督 | 数据导出、关联分析、自定义指标 |
| 情感驱动 | 焦虑与不甘 | 恐惧与怀念 | 掌控感与社交认可 |

## 3. 功能列表

### 3.1 核心功能（MVP必备）

| # | 功能 | 描述 |
|---|------|------|
| 1 | 习惯创建与编辑 | 允许用户自定义习惯名称、图标和颜色，打造个性化的追踪目标。 |
| 2 | 每日打卡签到 | 提供一键打卡操作，让用户快速记录当天是否完成习惯。 |
| 3 | 习惯频率设置 | 支持设置习惯的执行频率（每天、每周N次），灵活适配不同习惯节奏。 |
| 4 | 日历视图 | 以日历形式直观展示每个习惯的完成历史，帮助用户一目了然地回顾表现。 |
| 5 | 连续打卡记录（Streak） | 自动计算并展示当前连续完成天数，利用"不断链"心理驱动用户坚持。 |
| 6 | 基础数据统计 | 展示每个习惯的完成率、总完成次数等核心指标，让用户了解自己的进步。 |
| 7 | 提醒通知 | 允许用户为每个习惯设置自定义时间的推送提醒，防止遗忘。 |
| 8 | 习惯分类/分组 | 支持将习惯按类别（如健康、学习、工作）分组管理，保持列表整洁有序。 |
| 9 | 习惯归档与删除 | 允许用户归档不再追踪的习惯或彻底删除，保持数据清爽。 |
| 10 | 本地数据持久化 | 确保所有习惯数据在本地安全存储，即使离线也能正常使用。 |

### 3.2 创新/高级功能（未来版本）

| # | 功能 | 描述 |
|---|------|------|
| 1 | AI 智能洞察 | 基于用户打卡数据，AI自动分析习惯完成规律并给出个性化建议。 |
| 2 | 习惯堆叠（Habit Stacking） | 引导用户将新习惯绑定到已有习惯上，利用行为科学提升养成成功率。 |
| 3 | 动态难度调整 | 根据用户完成情况自动调整习惯目标难度，避免因目标过高而放弃或过低而无聊。 |
| 4 | 社交 accountability 伙伴 | 允许用户邀请好友互相监督打卡进度，通过社交压力和鼓励提升坚持率。 |
| 5 | 游戏化成就系统 | 解锁徽章、等级和虚拟奖励，将习惯养成变成一场有趣的成长游戏。 |
| 6 | 情绪与习惯关联分析 | 让用户记录每日情绪，分析习惯完成度与情绪状态之间的相关性。 |
| 7 | 习惯模板市场 | 提供由社区或专家创建的习惯计划模板，一键导入即可开始。 |
| 8 | 智能情境提醒 | 基于位置、天气或日程等情境自动触发提醒。 |
| 9 | 可视化热力图 | 以GitHub式热力图展示全年习惯完成密度，让长期进步一目了然。 |
| 10 | 数据导出与报告 | 支持将习惯数据导出为PDF/CSV，或生成精美的周报/月报。 |
| 11 | 习惯依赖链 | 允许设置习惯之间的前置依赖关系，构建正向行为链。 |
| 12 | Widget 小组件 | 提供桌面/锁屏小组件，无需打开App即可快速打卡和查看进度。 |
| 13 | 多设备云同步 | 通过云端同步确保多端数据实时一致。 |
| 14 | 深色模式与主题定制 | 提供深色模式及多种主题配色方案。 |
| 15 | 语音打卡 | 支持语音助手快速记录习惯完成状态。 |

## 4. 用户故事

### 4.1 林小禾的视角（21岁 · 压力很大的在校大学生）

> 她的关键词：全盘崩溃、缺乏反馈、同时想改太多

#### US-01 习惯创建与编辑

**作为林小禾**，我想要快速创建一个习惯并给它起个名字、选个颜色，以便我能把脑子里"想做的事"立刻变成一条具体记录，而不是在复杂的设置页面里花10分钟还没开始。

*场景*：晚上躺在床上，她想到"我明天开始要早睡"，希望30秒内就把这个念头变成一条可追踪的习惯。

#### US-02 每日打卡签到

**作为林小禾**，我想要在完成习惯后一键打卡，以便我能用最少的操作留下"我今天做到了"的记录，不需要填表、写日记或打开好几个页面。

*场景*：早上7点闹钟响了，她真的起床了，想立刻点一下确认"早起✓"，然后开始新的一天。

#### US-03 习惯频率设置

**作为林小禾**，我想要设置"每周完成4次"而不是只能选"每天"，以便我在某天因为赶作业没做到时，不会觉得整个计划已经失败，这周还有机会补回来。

*场景*：她给自己定了"每周运动4次"的目标，周二和周四因为期中复习没去跑步，但周一、周三、周五、周六去了——4次达标，她依然有成就感。

#### US-04 连续打卡记录（Streak）

**作为林小禾**，我想要看到自己连续坚持了多少天，以便当数字变成7、14、21时，我会舍不得让这个数字归零，从而在"算了今天不做了"的瞬间再撑一下。

*场景*：连续早起第12天，冬天的早晨她真的想赖床，但看到"12天连续"的数字，她咬牙坐了起来——她不想让12归零。

#### US-05 日历视图

**作为林小禾**，我想要在日历上看到每天习惯的完成情况，以便我能一眼看出"上周前三天做得很好，后四天全崩了"，找到自己的节奏规律，而不是只看到一个冷冰冰的百分比。

*场景*：周日晚上复盘时，她打开日历，发现每次周三都是空白——原来周三下午有固定的小组讨论，总是打乱她的节奏，她决定把周三的习惯时间调到早上。

#### US-06 基础数据统计

**作为林小禾**，我想要看到每个习惯的完成率和总完成次数，以便我在坚持了一段时间后能对自己说"你看，你这个月60%的日子都做到了，不是0%"，而不是只记得那些失败的日子。

*场景*：月底她心情低落，觉得"这个月又废了"，但打开统计看到"早起完成率58%"——虽然不完美，但确实比上个月好了。这个数字让她愿意再试一个月。

#### US-07 提醒通知

**作为林小禾**，我想要在习惯的执行时间收到推送提醒，以便我在沉浸学习或刷手机时被"拉回来"，不会因为忘记而错过，然后又陷入自责循环。

*场景*：下午2点她正在刷B站，手机弹出提醒"该背单词啦"——她差点忘了，看到提醒后放下手机打开了单词书。

#### US-08 习惯分类/分组

**作为林小禾**，我想要把习惯分成"学习""健康""生活"几组，以便我同时追踪6-7个习惯时不会觉得列表乱成一锅粥，而是能分块管理、逐个击破。

*场景*：她有7个习惯——早起、运动、背单词、冥想、少刷手机、11:30睡觉、喝2L水。没有分组时列表太长让她焦虑，分组后她觉得"先从健康类开始，其他的慢慢加"。

#### US-09 习惯归档与删除

**作为林小禾**，我想要把已经养成或不想再追踪的习惯归档，以便我的习惯列表保持精简，不会被"已经不需要追踪的习惯"干扰注意力。

*场景*：坚持了60天后，"每天刷牙"已经完全自动化，她把它归档，列表上只留下还在努力中的5个习惯，视觉负担减轻了。

#### US-10 本地数据持久化

**作为林小禾**，我想要我的所有习惯数据保存在手机本地，以便我在地铁上、图书馆里、甚至没网的时候都能正常打卡和查看记录，不会因为网络问题丢失数据。

*场景*：在图书馆地下室信号很差，但她仍然能打开App打卡"今日学习1小时"，数据不会丢失。

### 4.2 赵一鸣的视角（26岁 · 工作繁忙的年轻职场人）

> 他的关键词：时间碎片化、缺乏弹性、一个人坚持太难

#### US-11 习惯创建与编辑

**作为赵一鸣**，我想要快速创建习惯并设置一个简短备注，以便我在通勤地铁上就能花1分钟把"每周运动3次"加进追踪列表，而不需要到办公室打开电脑才能操作。

*场景*：早上地铁上，他看到一篇关于久坐危害的文章，立刻打开App添加了"每小时站起来活动5分钟"的习惯。

#### US-12 每日打卡签到

**作为赵一鸣**，我想要在锁屏或通知栏直接完成打卡，以便我在加班到很晚回家后，不需要再打开App翻好几页，就能快速确认"今天做了"。

*场景*：晚上10:30加完班到家，他确实在午休时做了20分钟拉伸，但已经累得不想再打开App——如果通知栏就能打卡，他会顺手点一下。

#### US-13 习惯频率设置

**作为赵一鸣**，我想要设置"每周3次"而非固定某几天，以便我周三加班时不用因为"计划是周一三五"而感到失败，而是可以周二、周四、周六补上，依然算达标。

*场景*：原计划周一三五健身，但周三临时被拉进项目赶工到10点。如果频率是"每周3次"，他可以周四和周六补上，这周依然完成目标——而不是觉得"周三没去，计划已废"。

#### US-14 连续打卡记录（Streak）

**作为赵一鸣**，我想要看到我连续多少周完成了"每周运动3次"的目标，以便当连续周数达到4周、8周时，我会觉得"我已经不是那个办了健身卡只去3次的人了"，这种身份认同的转变让我愿意继续。

*场景*：连续第6周完成"每周运动3次"，他截图发了朋友圈——6周前他还是那个"说去健身但从不去"的人，现在他开始认同"我是一个有运动习惯的人"。

#### US-15 日历视图

**作为赵一鸣**，我想要在日历上看到哪些天完成了习惯、哪些天没有，以便我能发现"我总是在周二和周四加班日跳过习惯"，从而主动把那两天的习惯时间调整到早上或午休。

*场景*：看了一个月的日历后，他发现周二和周四几乎总是空白——这两天是固定加班日。他决定把这两天的运动改到早上7点，而不是晚上。

#### US-16 基础数据统计

**作为赵一鸣**，我想要看到月度完成率的趋势变化，以便我能确认"我是不是真的在进步"，而不是凭感觉——因为凭感觉他总觉得"这周又没做好"，但数据可能告诉他"你这个月比上个月提升了15%"。

*场景*：月底他感觉"这个月又很糟糕"，但打开统计看到运动完成率从上月的40%升到了55%——进步确实存在，只是不够显著，他决定下个月目标定到65%。

#### US-17 提醒通知

**作为赵一鸣**，我想要在特定时间收到提醒，但更希望提醒能区分优先级，以便我不会被7个习惯的提醒同时轰炸而全部忽略，而是只收到"今天还没完成的最重要的2个"提醒。

*场景*：晚上8点，他只收到"今日未完成：运动、阅读30分钟"的提醒，而不是7个习惯全部弹一遍——后者只会让他烦躁地全部划掉。

#### US-18 习惯分类/分组

**作为赵一鸣**，我想要把习惯分成"运动""个人成长""健康"几组，以便我在时间有限时能优先关注最重要的"运动"组，而不是面对一个扁平的列表不知道该先做哪个。

*场景*：今晚只有30分钟自由时间，他打开App看到"运动"组标红（本周还差1次），果断选择去做20分钟HIIT，而不是漫无目的地在列表里纠结。

#### US-19 习惯归档与删除

**作为赵一鸣**，我想要归档已经稳定养成的习惯，以便我的注意力集中在"还在挣扎的习惯"上——就像工作里优先处理最紧急的任务一样。

*场景*：坚持3个月后，"每天喝2L水"已经成了本能，他把它归档。现在列表上只剩"每周运动3次"和"每天阅读30分钟"——聚焦，更容易坚持。

#### US-20 本地数据持久化

**作为赵一鸣**，我想要数据存储在本地且响应极快，以便我在公司电梯里、会议间隙等碎片时间打开App时，不需要等待加载就能立刻打卡，零摩擦。

*场景*：会议间隙有2分钟，他打开App秒开，快速打卡"午休拉伸"，整个过程不超过5秒。如果需要等3秒加载，他可能就不会打开了。

## 5. 范围外功能清单

> **原则**：MVP只做"让坚持被看见""让计划有弹性""让起步更简单"三件事。任何不直接服务于这三个核心价值、或实现成本过高的功能，均列入范围外。

### 5.1 全部高级/创新功能（MVP阶段不做）

| # | 功能 | 排除理由 |
|---|------|----------|
| 1 | AI 智能洞察 | 需要大量用户数据训练模型，MVP阶段数据量为零；技术复杂度远超MVP承受范围 |
| 2 | 习惯堆叠引导 | 需要设计引导流程和推荐逻辑，MVP阶段用户只需能创建习惯即可，堆叠理念可通过内容运营传达 |
| 3 | 动态难度调整 | 需要算法判断用户状态并自动调整目标，逻辑复杂且MVP阶段缺乏数据支撑决策 |
| 4 | 社交 accountability 伙伴 | 涉及用户体系、好友关系、消息系统，是一个完整社交模块，MVP无法承载 |
| 5 | 游戏化成就系统 | 需要设计徽章体系、等级规则、虚拟奖励，设计+开发量大，且Streak已提供基础游戏化动力 |
| 6 | 情绪与习惯关联分析 | 需要额外开发情绪记录模块和关联分析算法，超出MVP核心价值范围 |
| 7 | 习惯模板市场 | 需要内容生产、分类、搜索、用户评价等完整内容系统，MVP阶段用户可直接手动创建 |
| 8 | 智能情境提醒 | 依赖位置服务、日历读取、天气API等系统级权限和第三方集成，技术复杂度高 |
| 9 | 可视化热力图 | 日历视图已在MVP范围内，热力图是增强版可视化，可在后续版本作为日历视图的升级替代 |
| 10 | 数据导出与报告 | 主要服务于苏铭远类用户（深度数据需求者），非MVP目标用户核心需求 |
| 11 | 习惯依赖链 | 逻辑复杂（需处理循环依赖、条件判断等边界情况），且MVP用户未表达此需求 |
| 12 | 语音打卡 | 需要语音识别集成，且MVP的一键打卡已足够简单，语音并非必需 |

### 5.2 部分核心功能的简化/延后项

| # | 功能 | MVP范围内 | 范围外（延后） |
|---|------|-----------|----------------|
| 1 | 提醒通知 | 单一时间点推送提醒 | 智能优先级排序、未完成习惯聚合提醒 |
| 2 | 习惯频率设置 | 每天 / 每周N次 | 自定义周期（如"每3天1次"）、间隔模式 |
| 3 | 基础数据统计 | 完成率、总完成次数、当前Streak | 趋势折线图、周/月对比、按分类汇总 |
| 4 | 习惯创建与编辑 | 名称、图标、颜色、频率 | 自定义标签、附件（照片/笔记）、子习惯 |
| 5 | 日历视图 | 单个习惯的月度日历 | 全部习惯叠加视图、周视图、年视图 |
| 6 | 打卡操作 | 列表页一键打卡 | 滑动手势打卡、桌面Widget打卡、通知栏打卡 |

### 5.3 平台与技术相关范围外项

| # | 项目 | 排除理由 |
|---|------|----------|
| 1 | 多设备云同步 | 需要后端服务、用户账号体系、数据冲突解决策略，MVP阶段本地存储足够 |
| 2 | Widget 桌面小组件 | 依赖平台特定API，且需额外设计适配，MVP优先保证App内体验 |
| 3 | 深色模式与主题定制 | 视觉增强功能，不影响核心价值交付，MVP采用单一主题 |
| 4 | 多平台支持（Web/Pad） | MVP专注单平台（手机端），确保核心体验打磨到位后再扩展 |
| 5 | 多语言/国际化 | MVP面向中文用户，国际化在产品验证成功后再考虑 |
| 6 | 用户账号与登录系统 | MVP无需云同步和社交功能，暂不需要账号体系，降低注册摩擦 |
| 7 | 第三方数据接入（Apple Health等） | 需要平台审核和API对接，且主要服务于深度数据用户，非MVP目标 |

### 5.4 明确排除的产品方向

| # | 方向 | 排除理由 |
|---|------|----------|
| 1 | 社区/社交功能 | 社交是双刃剑——冷启动阶段没有内容也没有人，反而增加孤独感；且社交模块开发量巨大 |
| 2 | 付费/订阅模式 | MVP阶段目标是验证产品价值，不是商业化；引入付费会阻碍用户获取和反馈收集 |
| 3 | 内容/课程模块 | 习惯追踪器不是教育平台，MVP专注"追踪"本身，内容引导可通过运营手段补充 |
| 4 | 习惯教练/AI对话 | 类似AI聊天助手的功能，技术成本高且MVP阶段用户需要的是简单工具，不是对话伙伴 |

### 5.5 范围外功能的版本规划参考

| 优先级 | 版本 | 候选功能 | 理由 |
|--------|------|----------|------|
| P0 | v1.1 | Widget小组件、深色模式、热力图 | 用户感知强，实现成本可控 |
| P1 | v2.0 | 云同步、账号体系、习惯堆叠引导 | 解锁新场景 |
| P2 | v3.0+ | AI洞察、社交伙伴、成就系统 | 产品差异化 |

## 6. 关键成功指标

> **选择原则**：MVP阶段的核心假设是——"弹性计划+可视化反馈+极简操作，能帮助用户把习惯坚持下去"。因此，成功指标必须直接验证这个假设，而非衡量虚荣指标（如下载量）。

### 6.1 指标一：7日留存率（Day 7 Retention Rate）

**定义**：注册/首次使用后，第7天仍返回App的用户占比。

**目标基准**：≥ 30%

**为什么选它**：

7日留存是习惯追踪器MVP的生死线。习惯追踪器的价值完全依赖持续使用——用户如果3天后就不再打开App，说明产品没有提供足够的"回来理由"，核心价值假设直接被推翻。我们的目标用户（林小禾、赵一鸣）的共同特征是"每次坚持不到一周就放弃"——如果我们的App也无法让他们撑过7天，那我们和他们之前试过的手账、番茄钟没有任何区别。7天恰好是Streak机制的第一个"里程碑时刻"——如果用户在第7天看到自己的连续记录，仍然选择离开，说明可视化反馈的驱动力不足。

**衡量方式**：

```
Day 7 Retention = 第7天打开App的用户数 / 首次使用用户总数 × 100%
```

**预警信号**：

| 区间 | 判断 | 行动 |
|------|------|------|
| < 15% | 产品核心体验存在根本问题 | 回溯审视Onboarding流程和首次使用体验 |
| 15%-25% | 用户有初步兴趣但动力不足 | 优化Streak展示和提醒触达策略 |
| 25%-30% | 接近及格线 | 微调提醒文案、增加第3天/第5天的里程碑提示 |

### 6.2 指标二：习惯7天连续达成率（7-Day Streak Achievement Rate）

**定义**：用户创建的习惯中，达到7天连续打卡完成的比例。

**目标基准**：≥ 25%

**为什么选它**：

这是最直接验证核心价值假设的指标——我们的产品宗旨是"帮助用户从意愿到坚持"，而7天连续达成正是"坚持"的第一个可观测证据。它比留存率更深入一层：留存率只说明"用户回来了"，但7天Streak达成率说明"用户真的在坚持做习惯"——这才是产品的核心交付物。对林小禾而言，7天是她过去从未突破的"放弃临界点"；对赵一鸣而言，连续7天完成"每周3次运动"意味着2-3周的持续执行，这已经是行为的质变。这个指标还能间接验证"弹性频率"的价值——如果"每周N次"的习惯7天达成率显著高于"每天"类习惯，说明弹性机制确实在减少"全盘崩溃"。

**衡量方式**：

```
7-Day Streak Rate = 达到7天连续打卡的习惯数 / 已创建满7天的习惯总数 × 100%
```

**预警信号**：

| 区间 | 判断 | 行动 |
|------|------|------|
| < 10% | 产品未能帮助用户跨越"第一周放弃"的鸿沟 | 检查频率设置是否真的灵活、打卡操作是否足够简单、Streak展示是否足够醒目 |
| 10%-20% | 部分用户受益但不够普遍 | 分析哪些类型的习惯达成率高、哪些低，针对性优化 |
| > 40% | 核心价值假设被强验证 | 可考虑将Streak里程碑扩展到14天、21天 |

### 6.3 指标三：日均打卡完成率（Daily Check-in Completion Rate）

**定义**：每日应完成习惯中，实际完成打卡的比例。

**目标基准**：≥ 55%

**为什么选它**：

这个指标衡量的是用户每一天的执行质量——不是"你来了吗"（留存），也不是"你坚持了多久"（Streak），而是"你今天该做的做了多少"。55%看似不高，但对于林小禾和赵一鸣这类"反复失败"的用户，从0%到55%已经是巨大的进步——这正是我们在产品宗旨中承诺的"从0到1"的价值。它能捕捉到留存率和Streak无法反映的部分成功——比如赵一鸣今天有5个习惯，完成了3个，Streak可能断了，但60%的完成率说明他并非全盘放弃，产品仍在为他创造价值。这个指标对弹性频率的验证尤为关键：如果"每周3次"类习惯的日完成率远高于"每天"类习惯，就证明了"弹性计划减少放弃"的假设。

**衡量方式**：

```
Daily Completion Rate = 当日实际打卡数 / 当日应完成习惯总数 × 100%
```
*按用户取日均值，再取全体用户的中位数。*

**预警信号**：

| 区间 | 判断 | 行动 |
|------|------|------|
| < 30% | 用户创建了习惯但不执行 | 检查：①创建时是否过于乐观；②提醒未有效触达；③打卡操作仍有摩擦 |
| 30%-45% | 用户在尝试但成功率偏低 | 分析是特定时段完成率低还是特定类型习惯完成率低，针对性优化 |
| > 70% | 用户执行良好 | 可考虑在后续版本引入更高阶的挑战（如动态难度调整） |

### 6.4 三个指标的协同关系

```
7日留存率 ──→ 习惯7天连续达成率 ──→ 日均打卡完成率
  "用户回来吗"     "用户坚持了吗"        "用户每天做得怎样"
   (门槛指标)        (核心指标)           (质量指标)
```

| 指标 | 回答的问题 | 验证的假设 | 失败意味着 |
|------|-----------|-----------|-----------|
| 7日留存率 | 用户愿意回来吗？ | 产品提供了"回来"的理由 | 产品未被感知到价值 |
| 7天连续达成率 | 用户真的在坚持吗？ | 弹性+可视化能帮助坚持 | 核心价值假设不成立 |
| 日均打卡完成率 | 每天的执行质量如何？ | 极简操作降低了执行摩擦 | 产品体验仍有障碍 |

**关键原则**：三个指标必须联合解读。

- 高留存 + 低Streak = 用户回来但没坚持 → 需要强化动力机制
- 低留存 + 高Streak = 少数铁粉在用 → 需要降低门槛扩大受众
- 高留存 + 高Streak + 低完成率 = 用户想坚持但执行困难 → 需要优化弹性频率和提醒策略

## 附录

### A. 产品经理补充建议

1. **MVP优先级建议**：核心功能1-6构成最精简的MVP闭环——能创建、能打卡、能回顾。提醒通知（7）和分类（8）紧随其后。

2. **差异化切入点**：在高级功能中，AI智能洞察和习惯堆叠是最具差异化的两个方向——前者提供"别人给不了的洞察"，后者基于《原子习惯》等畅销书理念，有天然的传播力。

3. **防流失设计**：Streak是习惯类App最核心的留存机制，建议在MVP阶段就加入"断链保护"（如每月允许1-2次补签），避免用户因偶尔中断而彻底放弃。
